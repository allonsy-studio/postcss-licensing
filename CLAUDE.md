# CLAUDE.md

Guidance for working in this repository.

## Project overview

`@allons-y/postcss-licensing` is a single-file PostCSS plugin that prepends a
license/copyright comment (`/*! ... */`) to CSS files. It reads the comment
content from a `COPYRIGHT` or `LICENSE` file (configurable via `filename` and
`cwd` options) and warns instead of failing when no file is found.

- **ESM only** (`"type": "module"`) — use `import`/`export`, never `require`.
- **Node >= 24**, **Yarn 4** via corepack (`yarn`, not npm/pnpm).
- `postcss` is a **peer dependency** — never move it to `dependencies`.

## Layout

| Path        | Purpose                                                 |
| ----------- | ------------------------------------------------------- |
| `index.js`  | The entire plugin. Exports default plugin + `messages`. |
| `test.js`   | AVA tests.                                              |
| `fixtures/` | Input CSS (and custom copyright files) for tests.       |
| `expected/` | Expected CSS output, compared byte-for-byte.            |

## Commands

```sh
yarn install     # install deps
yarn test        # run AVA tests
yarn coverage    # tests with c8 coverage
yarn lint        # eslint check
yarn format      # eslint --fix
```

Husky + lint-staged run eslint/prettier on commit; don't bypass with `--no-verify`.

## Testing

Tests compare `fixtures/<name>.css` processed by the plugin against
`expected/<name>.css` — an exact string match, including whitespace and the
trailing newline. For new behavior: add a fixture, add an expected file, and
add a test in `test.js` using the existing `compare()`/`fetchWarnings()`
helpers. Warning text must match `messages.fileNotFound()` exactly.

## Releases and changesets

Releases are automated by **Changesets** (the release workflow opens a
"Version Packages" PR). For any change that affects consumers, run
`yarn changeset`, pick the bump (patch = fix, minor = feature,
major = breaking), and commit the generated `.changeset/*.md` file with your
change. The changeset summary becomes the CHANGELOG entry — write it for a
human reader. Docs/CI/test-only changes need no changeset. Never edit
`CHANGELOG.md` or the `version` field in `package.json` by hand.

## Commit messages

Use Conventional Commits:

```text
<type>(<optional-scope>): <imperative subject, ≤100 chars, lowercase, no period>

<body: what changed and why, in plain prose, no-wrap, 1-2 sentences <80 chars>

Closes #N
```

- `<type>`: feat, fix, docs, chore, refactor, test, …
- Subject in imperative mood ("add", not "added").
- Small commits may be subject-only; write a body when the change needs context.
- Do NOT add "Generated with Claude Code" or Co-Authored-By footers.

## Pull requests

- Branch from `main`: `fix/short-description`, `feat/short-description`,
  `docs/short-description`.
- One logical change per PR; reference the issue (`Closes #123`).
- Lint and test locally before opening; include a changeset when required
  (see above).
- PR title follows the same conventional-commit format as commit subjects.

## Code style

- **DO:** keep code self-documenting. When a comment is warranted, keep it
  brief and explain only the _why_ that the code can't show — never restate
  what the code does.
- README sections between `<!-- weaver:*:START -->` / `<!-- weaver:*:END -->`
  markers are auto-generated by Weaver — never edit inside them.
- Match the existing style in `index.js` (JSDoc types, double quotes,
  formatting is enforced by eslint/prettier).
